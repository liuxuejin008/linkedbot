1.proxy模式，外部系统回调channel的webhook的地址的时候，channel透传回去本地client，然后Forwarded to localhost:9999/webhook，获得

我重新理清楚LinkedBot的架构。

1.有server端，跑在云上或者有公网IP的服务器上，我们把server端叫做ChannelServer

2.有客户端，跑在办公室的电脑上，或者办公室的内网上，没有公网地址，我们把客户端叫做ChannelClient，他一直用长连接和ChannelServer保持通信，
通过sse协议接收ChannelServer的消息


3.本地的WebHook接收端，我们跑在办公室的电脑上，或者办公室的内网上，没有公网地址，我们叫Channel Receiver。

架构要点说明：
ChannelServer (公网端)：
角色：消息中转站（Relay）。
职责：提供公网 Endpoint 接收第三方回调；维护与 ChannelClient 的 SSE 连接池；将收到的 HTTP Body 转换为 SSE Event 发送。
ChannelClient (内网客户端)：
角色：内网穿透代理（Agent）。
职责：主动向 Server 发起连接（避免了内网防火墙拦截入站流量的问题）；解析 SSE 事件流；将数据重新构造为本地 HTTP 请求发给 ChannelReceiver。
ChannelReceiver (本地接收端)：
角色：最终业务逻辑处理。
职责：运行在内网，处理具体的业务逻辑（如：解析告警、自动部署、控制局域网设备等）。

    Note right of CS: 生成请求 ID (ReqID)<br/>挂起当前 HTTP 连接 (Hold)

    CS-->>CC: 2. 通过 SSE 推送数据 (包含 ReqID + 载荷)
    
    activate CC
    CC->>CR: 3. 转发本地请求 (POST /local)
    activate CR
    CR-->>CC: 4. 返回处理结果 (Result JSON)
    deactivate CR

    CC->>CS: 5. 提交处理结果 (POST /callback {ReqID, Result})
    deactivate CC

    Note right of CS: 根据 ReqID 匹配挂起的请求
    CS-->>Web: 6. 同步返回处理结果 (200 OK + Result)
    deactivate CS

架构关键点补充：
ReqID (关联 ID)：
ChannelServer 在接收到外部 Webhook 时，必须生成一个唯一的 ReqID，并把它随 SSE 消息下发。没有这个 ID，后续 ChannelClient 回传结果时，Server 无法知道这个结果对应的是哪一个正在挂起的外部请求。
同步挂起 (Request Parking)：
ChannelServer 的 Webhook 入口函数不能立即结束。它需要使用类似 Promise、Future 或 Channel (Go语言) 的机制在内存中“等一下”，直到接收到对应的回调。
超时机制：
由于增加了内网传输环节，必须设置超时（比如 10-30秒）。如果 ChannelClient 在时间内没有回传结果，ChannelServer 应该给外部系统返回一个 504 Gateway Timeout。
反向回传 (Return Path)：
ChannelClient 返回结果给 ChannelServer 时，是发起一个新的 HTTP POST 请求到 ChannelServer 的特定 API 接口（例如 /api/callback）。这是因为 SSE 是单向的，不能直接反向写数据。
